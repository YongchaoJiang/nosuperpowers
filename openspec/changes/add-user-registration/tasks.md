## 2. 注册路由与表单

- [ ] 2.1 新增 `views/register.ejs`，包含邮箱、密码输入框、提交按钮，以及用于展示校验错误信息的位置。
- [ ] 2.2 在 `server.js` 中新增 `GET /register` 路由，渲染新模板。
- [ ] 2.3 在 `POST /register` 处理函数中加入邮箱格式校验（拒绝缺失/格式错误的邮箱，重新渲染表单并展示错误）。
- [ ] 2.4 在 `POST /register` 处理函数中加入密码最小长度校验（拒绝缺失/过短的密码，重新渲染表单并展示错误）。
- [ ] 2.5 在创建记录前，对 `users` 集合做邮箱重复校验（用统一的"该邮箱已被注册"错误提示拒绝，重新渲染表单）。
- [ ] 2.6 在写入新用户记录之前，用 `bcryptjs`（和管理员登录用的同一套 API）对密码做哈希。
- [ ] 2.7 每条新创建的记录都显式设置 `role: 'user'`。
- [ ] 2.8 在 `POST /register` 处理函数中调用现有的 `isRateLimited`/`recordSubmission` 辅助函数（目前用于评论功能），以请求 IP 为 key，命中限流时用"请稍后再试"提示拒绝。
- [ ] 2.9 创建成功后，参照 `POST /admin/login` 的方式设置 `req.session.userId`（以及 `role`），让新访客立即处于已登录状态。

## 3. 通用登录路由

- [ ] 3.1 改造 `views/login.ejs` 的文案，去掉"管理员"专属措辞，改成面向所有账号类型的通用登录页。
- [ ] 3.2 改造现有 `POST /admin/login` 处理函数：查询逻辑从"只按 `username` 查管理员"扩展为同时支持公开用户按邮箱登录（凭证不匹配时统一提示"账号或密码错误"，不泄露是账号不存在还是密码错误）。
- [ ] 3.3 登录成功后按查到的 `role` 分流跳转：`role === 'admin'` 跳 `/admin`，否则跳回首页 `/`。
- [ ] 3.4 在 `POST /admin/login` 里调用现有的 `isRateLimited`/`recordSubmission`（和注册共用同一份限流状态），命中限流时提示稍后再试，不校验凭证。

## 4. 访问控制
