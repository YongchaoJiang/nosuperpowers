## Why

`add-user-registration` 让访客能注册并登录，但注册完的账号除了"能登录"以外什么也做不了——没有任何功能会消费这个身份。这个 change 让注册用户拥有真正的用途：发布自己的博客文章、管理自己文章下的评论；同时让管理员从"唯一的内容生产者"转变为"平台的全局管理者"，可以监管所有用户的文章和评论。

## What Changes

- 给 `articles` 增加归属字段（`authorId`），文章不再隐含"只属于唯一的管理员"。
- 注册用户可以在新增的 `/dashboard` 下自由创建、编辑、发布/取消发布、删除**自己的**文章，不需要管理员审批即可对外发布（自由发布模型）。
- 注册用户可以在 `/dashboard/comments` 审核（通过/拒绝）**自己文章下**的评论，权限逻辑限定在自己拥有的文章范围内。
- 管理员保留对所有用户文章和评论的完全管理权：`/admin` 下可以查看、编辑、删除任意用户的文章，`/admin/comments` 可以审核任意文章下的评论。
- **BREAKING**：新增管理员"下架"能力——管理员可以强制取消发布任意文章，被下架的文章 SHALL 无法被文章作者自行重新发布，只有管理员可以恢复发布。这改变了"发布/取消发布是文章作者独占权限"的现状。
- 原有的文章 CRUD 路由（`/admin/articles/...`）迁移到 `/dashboard/articles/...`，权限从"任意已登录管理员"改为"文章作者本人或管理员"；`/admin` 保留下来，但含义从"我的文章管理后台"变成"全局监管视图"。
- **BREAKING**：评论提交从"匿名+自由填写昵称"改为必须登录，评论记录关联到提交者的账号身份；提交表单不再有昵称/邮箱输入框。
- 评论的审核状态从"通过/删除"两种操作，扩展为"通过/拒绝/删除"三种，其中"拒绝"是可逆的状态（不同于永久性的删除）。
- **BREAKING**：新增管理员"覆盖评论审核结果"的能力——即使文章作者已经把某条评论审核通过，管理员仍可以将其重新置为拒绝；只要是管理员执行的拒绝，SHALL 只有管理员能重新批准，文章作者本人无权恢复。这和文章下架/恢复是同一套"作者可自治、管理员可否决且否决具有排他恢复权"的模式。
- 注册用户可以在 `/dashboard/comments` 里查看自己**发表过**的所有评论（不论发表在谁的文章下），并删除其中任意一条；这是和"审核自己文章下的评论"完全不同的另一重身份（评论的作者 vs 文章的作者），两者在同一个页面里以不同的视图/标签呈现。用户不能借此自行通过（approve）自己发表的评论——删除权和审核权分开授予，避免自我批准绕开审核。

## Capabilities

### New Capabilities
- `article-authorship`：文章归属与所有权模型——文章作者可以创建、编辑、删除自己的文章；管理员可以对任意用户的文章执行相同操作。
- `article-publish-moderation`：发布状态的管理员强制介入——文章作者可以自由发布/取消发布自己的文章，但管理员可以强制下架任意文章，下架后只有管理员能恢复发布，作者本人无法自行恢复。
- `comment-authorship`：评论提交要求登录，评论记录关联到提交者的账号身份，公开展示的身份来自账号信息而不是自由填写的昵称文本。
- `comment-moderation-scope`：评论审核的权限范围——普通用户只能审核（通过/拒绝/删除）自己文章下的评论；管理员可以审核所有文章下的评论，不受归属限制。
- `comment-moderation-override`：评论审核结果的管理员强制介入——管理员可以覆盖任意评论（包括作者已批准的）的审核结果；管理员执行的拒绝会锁定该评论，只有管理员可以重新批准，文章作者无权覆盖管理员的决定。这和 `article-publish-moderation` 是同一套模式在评论上的镜像。
- `comment-self-management`：用户查看和删除自己发表过的评论——不论这条评论发表在谁的文章下、当前处于什么审核状态，评论的原作者都可以看到并删除它，但不能借此自行批准（approve）自己的评论。

### Modified Capabilities
（无——`add-user-registration` 目前尚未实施/归档，`openspec/specs/` 里还没有 `user-login` 等能力的正式 spec 可供修改。等 `add-user-registration` 完成并归档后，本 change 会需要对 `user-login` 的"登录后跳转目标"这条需求做一次小改动：普通用户登录成功后应该跳转到 `/dashboard` 而不是首页。这个改动到时候要在 `add-user-registration` 归档之后，作为一个正式的 Modified Capability 补上，现在还不能创建这个 delta。）

## Impact

- **前置依赖**：本 change 依赖 `add-user-registration` 已经实施完成——需要 `users` 集合上的 `role` 字段、注册流程、以及通用登录能力才能开始（评论提交现在也需要登录，直接复用同一套会话机制）。
- **代码**：`server.js` 新增 `authorId`、`adminSuspended` 等文章字段；把现有 `/admin/articles/*`、`/admin/comments/*` 相关逻辑拆分成 `/dashboard/*`（自身范围）和 `/admin/*`（全局范围）两组路由；新增 owner-or-admin 权限判断（区别于现有的纯 `requireAuth`）；新增管理员下架/恢复路由；评论提交路由改为要求登录，评论记录新增 `userId`、`adminLocked` 字段，`status` 增加 `rejected` 取值。
- **数据**：`blog.json` 里现有的文章记录（目前只有管理员发布的文章）需要一次性迁移，补上 `authorId` 指向管理员账号，否则老文章会成为"孤儿数据"；现有的匿名评论记录（`nickname`/`email` 字段，没有 `userId`）也需要一次性处理，见 design.md 的 Migration Plan。
- **视图**：`views/admin/dashboard.ejs` 等现有管理后台视图需要拆分或复制出一份面向普通用户的 `/dashboard` 版本；`/admin` 下的文章列表需要新增"作者"列和下架/恢复操作入口；评论提交表单去掉昵称/邮箱输入框；评论审核界面新增"拒绝"操作和锁定状态提示；`/dashboard/comments` 新增"我发表的评论"视图/标签，与原有的"我文章下的评论"视图并存。
- **依赖**：无新增 npm 包——延续项目一贯的依赖克制原则。
