## Context

这个应用是单文件的 Express 服务（`server.js`），数据存储用 `lowdb`（JSON 文件 `blog.json`，没有真正的数据库/ORM）。目前已经有一个启动时种子生成的管理员账号，用 `bcryptjs` + `express-session` 做认证（`server.js:3,26-35,45-50,215-227`），还有一个基于内存的 `isRateLimited`/`recordSubmission` 限流器（`server.js:78-89`），目前只用在评论提交上。项目没有 route/model/validation 的目录结构——所有逻辑都在 `server.js` 里，按注释分区组织，视图是服务端渲染的 EJS 模板。

这次设计要在 `users` 集合里加入第二类账号（基于邮箱的公开账号），而目前这个集合隐含假设"只会有一条管理员记录"。

## Goals / Non-Goals

**Goals（目标）：**
- 让访客可以通过服务端渲染的表单，用邮箱+密码创建账号。
- 让注册用户之后能用同一套凭证重新登录，不只是注册那一刻的自动登录。
- 让新账号类型和管理员账号明确区分开，确保现有的管理员专属路由仍然受到保护。
- 复用已有模式（bcryptjs 哈希、express-session、内存限流），不引入新的库或架构。

**Non-Goals（非目标）：**
- 密码重置、邮箱验证/确认，以及任何形式的邮件发送。
- 任何消费账号身份的功能（例如发布文章、按归属审核评论、用户中心页面）——这些属于依赖本变更的另一个 change（角色权限/多作者能力），不在这里实现。
- 迁移出 lowdb/JSON 存储模型——本变更在现有存储模型内完成。

## Decisions

**用 `users` 记录上的 `role` 字段区分管理员和公开用户，而不是新建一个独立集合。**
考虑过的替代方案：新建一个和 `users` 分开的 `publicUsers` 集合。放弃的原因是 `requireAuth` 和会话结构已经以 `users` + `req.session.userId` 为核心；拆成两个集合需要重复或分叉这部分逻辑。在现有 schema 上加一个 `role: 'admin' | 'user'` 字段改动更小，也能保持"谁能登录"这件事只有一个数据源。`requireAuth`（管理后台路由）需要更新为同时检查 `role === 'admin'`，而不只是检查 `userId` 是否存在，否则公开用户的会话会满足这个检查条件。

**校验和哈希逻辑直接写在路由处理函数里，和评论路由的风格保持一致。**
考虑过的替代方案：抽出一个 `validators.js`/`models/user.js` 模块。这次没有采用，因为代码库里没有这种结构的先例（评论校验就是直接内联在 `server.js:167-171`），引入一种新的文件组织方式属于和本变更无关的更大改动。如果未来有更多认证相关功能加入，再抽出共享模块会更值得。

**`/register` 直接复用现有的基于 IP 的 `isRateLimited`/`recordSubmission` 辅助函数，不做修改。**
考虑过的替代方案：单独做一个按邮箱维度限流的机制，防止同一 IP 针对大量邮箱做时序探测（enumeration-by-timing）。目前没有采用——现有的辅助函数已经按 IP 限流，对第一版来说"够用"；本提案的范围是注册功能本身，而不是防滥用加固。已在下方"Open Questions"中记录为待定问题。

**注册成功后，参照 `POST /admin/login` 的写法，直接设置 `req.session.userId`/`role` 完成自动登录。**
考虑过的替代方案：注册后跳转到登录页面，让用户自己再登一次。放弃的原因是多此一举——账号刚创建、密码刚校验过，没有必要让用户马上再输一次。

**把 `GET /admin/login` / `POST /admin/login` 直接改造成通用登录路由，供管理员和公开用户共用，而不是新建一条独立的 `/login` 路由。**
考虑过的替代方案 1：保留 `/admin/login` 只给管理员用，另建 `/login` 给公开用户。放弃的原因是这会让"登录"这件事有两条平行路径、两份几乎一样的校验逻辑（都是查 `users`、`bcrypt.compareSync`、设置 `req.session`），后续任何一处改动都要同步改两份，容易出现两个入口行为不一致的安全隐患。
考虑过的替代方案 2：保留路径为 `/admin/login`，但改名成中性的 `/login`（同时保留旧路径重定向）。这其实是更干净的做法，但改路径涉及到现有书签、`res.redirect('/admin/login')` 等多处引用，是一个和"让登录通用化"这个核心目标无关的额外改动。为了让这次 change 保持聚焦，先复用现有路径不改名；路径重命名可以留给后续变更（比如做用户中心页面重构时）一并处理。
统一路由内部按查到的用户 `role` 分流：`role === 'admin'` 跳 `/admin`，否则跳回首页 `/`（用户中心尚不存在，等依赖本变更的下一个 change 引入 `/dashboard` 后，再把这个跳转目标改过去——这是一个明确的、留给下一个 change 处理的收尾点，不是遗漏）。

**登录尝试复用现有的基于 IP 的 `isRateLimited`/`recordSubmission` 辅助函数，和注册共用同一份限流状态。**
考虑过的替代方案：给登录单独维护一个基于「邮箱/用户名」维度的限流器，防止针对单个账号的密码暴力破解（IP 限流防不住换 IP 或者用代理池的攻击）。这次没有采用——现有辅助函数是纯 IP 维度、进程内存态的，追求"够用就好"；按账号维度限流是更精细的防护，值得在后续看到实际暴力破解风险时再加，已记录到 Open Questions。

**给老的管理员记录做启动时自动补丁，显式写入 `role: 'admin'`，而不是在 `requireAuth` 里永久挂一条"缺失 `role` 就当 admin"的兜底判断。**
考虑过的替代方案：防御式默认值——`requireAuth` 用 `user.role || 'admin'` 兜底，永远不改 `blog.json` 里的老数据。放弃的原因是这会让 `users` 集合永久混着两种数据形状（老记录没有 `role` 字段、新记录有），`blog.json` 从此不再自解释，且"缺失即 admin"这条隐藏规则要被后续每一处读取 `role` 的代码记住，容易在新代码里漏掉而埋下权限判断的坑。项目里已经有现成的先例可以照搬：`nextCommentId` 的补丁逻辑（`server.js:20-23`）就是在启动时检查字段是否存在、不存在就写入默认值。给管理员记录的 `role` 字段用同样的模式做一次幂等补丁，成本和防御式默认值几乎一样低，但补丁完成后 `requireAuth` 可以直接写 `role === 'admin'`，不需要永久保留 `|| 'admin'` 这层兜底。

## Risks / Trade-offs

- [用户数据通过 lowdb 以明文 JSON 存储] → 密码在写入前用 bcryptjs 哈希，和现在的管理员账号做法一致；磁盘上不会出现明文密码。邮箱地址仍以明文存在 `blog.json` 中，这和应用现在存储其他数据的方式一致（考虑到当前的威胁模型可以接受）。
- [没有邮箱验证] → 任何人都可以用不属于自己的邮箱注册。这在本变更范围内可以接受，已在 Non-Goals 中标注，属于自然的后续工作。
- [基于 IP 的限流是共享代码，没有针对注册场景单独调优] → 同一 IP 上的正常评论流量和注册流量可能互相影响。考虑到目前的流量规模（单文件 JSON 存储的博客），风险较低；如果观察到滥用行为再重新评估。
- [给 `users` 加 `role` 字段改变了隐含的 schema] → 通过启动时的幂等补丁自动给老的管理员记录补上 `role: 'admin'`（见下方 Migration Plan），确保升级后 `requireAuth` 不会因此对现有管理员登录失效，同时不需要维护永久的兜底逻辑。

## Migration Plan

1. 在种子逻辑那块（`server.js:26-35` 附近），仿照 `nextCommentId` 的补丁写法（`server.js:20-23`），新增一段启动时检查：如果 `username: 'admin'` 的记录存在且没有 `role` 字段，就用 `.assign({ role: 'admin' }).write()` 给它补上。这段检查是幂等的，每次启动都能安全重复执行。
2. 新注册的账号一律显式写入 `role: 'user'`。
3. `requireAuth`（以及任何判断角色的地方）直接用 `role === 'admin'` 做比较，不需要 `user.role || 'admin'` 这种兜底写法——补丁逻辑保证了 `users` 集合里的每条记录迟早都会有显式的 `role` 字段。
4. 没有需要担心的回滚数据问题：这次改动对 schema 是增量式的（新增字段、新增路由）；回滚代码后，`blog.json` 里的 `role` 字段只是被闲置，不影响其他功能。

## Open Questions

- 注册/登录频率限制除了按 IP，是否还要按邮箱/账号维度限流，以减缓针对单个账号的暴力破解和对已注册邮箱的枚举探测？暂缓——目前的辅助函数只按 IP 限流。
- 密码的最小长度/复杂度要求是多少？本设计假设采用简单的最小长度校验（例如 8 位），与应用其他地方低复杂度的校验风格保持一致；具体阈值在实现阶段再确认。
- 登录路径要不要从 `/admin/login` 改名成中性的 `/login`？本设计决定先不改（见上方 Decisions），但下一个 change 引入用户中心时可能是重新考虑这个问题的好时机。
